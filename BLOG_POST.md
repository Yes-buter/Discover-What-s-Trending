但我不想写枯燥的爬虫代码，也不想处理繁琐的前后端联调。于是，我决定挑战一下：**完全不写一行代码（No-Code），只用自然语言（Prompt），让 AI 帮我把这个网站做出来。**

## 4. 需求拆解与 Prompt 迭代实战

在开发过程中，并非一帆风顺。这里分享几个关键节点的 Prompt 设计，展示 AI 是如何帮我解决具体问题的。

### 4.1 核心功能开发：从报错到修复

在实现“收藏”功能时，我遇到了一个棘手的 Bug。注册用户后，点击收藏 GitHub 项目，后端直接报 400 错误。

**我的 Prompt：**
```text
Signup error: AxiosError: Request failed with status code 400
```

**AI 的分析与操作：**
Trae 迅速分析了后端日志和数据库 Schema。它发现 `favorites` 表的 `item_id` 字段定义为 `UUID` 类型，但 GitHub 的项目 ID 是 `Integer` 类型（如 `123456`）。类型不匹配导致写入失败。

**AI 自动生成的修复方案（无需我动手）：**
它直接帮我创建了一个新的 SQL 迁移文件，修改字段类型：

```sql
-- supabase/migrations/20240131000003_change_favorites_item_id_type.sql
ALTER TABLE "public"."favorites" ALTER COLUMN "item_id" TYPE text;
```

这就是 **"零代码"** 的核心体验：我负责描述问题，AI 负责定位并写代码修复。

### 4.2 路由与页面缺失修复

当我兴冲冲地打开 Profile 页面时，却显示 404。

**我的 Prompt：**
```text
http://localhost:5173/profile 打开后 Page not found，并且收藏 pdf 报错。
我收藏了的 github 项目或者论文理应在 http://localhost:5173/profile 里展示才对。
```

**AI 的操作：**
1.  **检测路由**：发现 `App.tsx` 中确实漏掉了 `/profile` 路由。
2.  **创建页面**：自动生成了 `src/pages/Profile.tsx`。
3.  **实现逻辑**：在 Profile 页面中自动添加了获取用户收藏列表的逻辑，并复用了之前的卡片组件来展示收藏的 GitHub 项目和论文。

### 4.3 部署与环境适配

本地跑通后，我要部署到 Vercel。

**我的 Prompt：**
```text
先把这个项目部署到 vercel 上吧。
```

AI 帮我生成了 `vercel.json` 配置文件。但在部署后，我又遇到了连接错误 `[Errno 111] Connection refused`。

**我的 Prompt：**
```text
我在部署后的网站进行注册提示 [Errno 111] Connection refused，登录已有账号提示 Invalid username or password
```

**AI 的深度排查：**
AI 意识到我在本地开发时为了科学上网，在代码里硬编码了代理配置（`127.0.0.1:7890`）。但这在 Vercel 的云端环境中是无法访问的。
AI 迅速修改了 `api/core/database.py`，添加了环境判断逻辑：

```python
# 仅在非 Vercel 环境下且未设置代理时，才使用本地代理
if os.environ.get("VERCEL") != "1" and not os.environ.get("HTTP_PROXY"):
    proxy_url = "http://127.0.0.1:7890"
    os.environ["HTTP_PROXY"] = proxy_url
    os.environ["HTTPS_PROXY"] = proxy_url
```

这波操作直接解决了线上环境的网络问题。

## 7. 踩坑总结

回顾这 2 小时的“零代码”旅程，虽然 AI 很强，但也需要我们具备基本的 **Review 能力**：

1.  **数据类型敏感度**：数据库设计时，ID 类型（UUID vs Int/Text）是常见坑点。
2.  **环境差异**：本地开发环境（Localhost）和云端生产环境（Production）的网络、环境变量往往不同，代码中要做适配。
3.  **Prompt 技巧**：报错时，直接把错误日志丢给 AI，通常比自己描述现象更有效。
